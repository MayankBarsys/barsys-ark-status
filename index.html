Section 01
What We're Building and Why It Matters
The Barsys Ark app connects to three cocktail-making hardware devices (Barsys 360, Shaker Pro, Coaster 2.0) via Bluetooth. Users discover recipes, get AI-powered recommendations from BarBot, and craft cocktails with precision hardware assistance.
This sprint's single mission: take the existing backend + AI + designs and wire them into a shippable iOS app. The screens exist (1,046 in the codebase). The APIs exist (95 endpoints live). The designs exist (7 of 10 flows). The job is INTEGRATION — connecting existing pieces, not building from scratch.

Section 07
Operating Rules
Non-negotiable rules for this sprint. Violations escalate immediately.
📦
Daily PR
At least 1 PR merged by 10 PM ET every day. GitHub bot flags violations. No exceptions.
👁️
PR Review SLA
4-hour review SLA. Raghav merges to develop. CEO merges to main.
🚨
Blocker SLA
2 hours max. Post in #barsys-ark. Tag @Raghav + @Mayank. If unresolved → CEO.
⏱️
1-Day Rule
Any task exceeding 1 day triggers automatic escalation from Mayank to Akshet. No grace period.
🔒
Scope Freeze
After Day 1 standup: no new features. Period. Only bug fixes and wiring work. Feature requests go to V5.
📋
Evidence Required
PR link, screen recording, or curl output. "I did it" without evidence = not shipped. Every claim verified.
🗓️
Daily Standup
10 AM ET daily. Format: SHIPPED: [PR link]. SHIPPING: [task + ETA]. BLOCKED: [issue + who owns it].
📊
No Hierarchy Bypass
Questions → check the plan → Mayank → Raghav. Never skip to CEO directly. Chain of command enforced.
⚡
CEO Review SLA
4-hour PR review on main. Not 4 days. This is ship week. Responsiveness is mandatory.

Addendum
V4.1 — Ingredient Architecture
The Ingredient Architecture spec (29 pages) and PhotoScanner v2 ULTRA spec (51 pages) introduce a unified data model that requires targeted changes to the V4 sprint plan. This addendum defines the additional backend, AI, and iOS work — integrated into the existing day-by-day timeline.
Why This Addendum
Two new specifications landed after the V4 plan was finalized:
  • › Ingredient Architecture (29 pages) — defines the Category → Product → Recipe data model
  • › PhotoScanner v2 ULTRA (51 pages) — adds camera-based bottle recognition mapped to the same model
These specs don't replace the V4 plan — they extend it with 9 new tasks and 3 modified iOS days.
Three-Layer Model
The foundational data architecture that bridges recipes and user inventory
Layer 1
Categories
What recipes reference  ·  ~50-80 types
"Vodka", "Triple Sec", "Lime Juice"

The Bridge
Product.category_id → Category.id
Every product maps to exactly one category
↕
Layer 2
Products
What users own  ·  specific bottles
"Absolut Citron", "Cointreau 750ml"
"Recipes speak in categories. Users own products. The system translates invisibly."
